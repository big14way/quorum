# Quorum — the agent proposes, the multisig disposes

**One-page submission summary.** Suite of three ZeroClaw WIT tool plugins +
one published crate. Custody tiers: T1 (squads-propose) and T0 (squads-watch,
tx-xray). Draft PR: [zeroclaw-plugins#97](https://github.com/zeroclaw-labs/zeroclaw-plugins/pull/97).
Core: [quorum-squads-core on crates.io](https://crates.io/crates/quorum-squads-core).
Source: [github.com/big14way/quorum](https://github.com/big14way/quorum).
Demo video: _link on submission_.

## The problem, restated honestly

Every agent-payments design answers one question: how do you let an untrusted
text generator direct money? Session keys answer with caps the model can try
to talk around. Durable nonces answer by keeping fragile transactions alive
longer. Both leave the agent holding spend authority.

## The answer: structure, not vigilance

Quorum makes the agent a **proposer that can never spend**. It files payment
proposals into a Squads v4 multisig; M-of-N humans approve on their phones;
the program executes with a fresh blockhash whenever they're ready.

- **Prompt injection cannot spend.** Proposing is not spending. In hardened
  mode the agent's key holds the Squads *Initiate-only* permission — the
  program itself makes voting or executing mathematically impossible, so
  total key theft yields spam proposals at worst.
- **Blockhash expiry disappears.** The proposal is the durable object.
  Durable nonces treat the symptom; proposals remove the disease.
- **Humans approve reality, not claims.** `tx-xray` renders any unsigned
  transaction into a capped receipt derived from the bytes plus an RPC
  simulation. Approve what it *does*, not what the agent *says*.

## Fail closed, provably

No allowlist or address book in config → refuses to run. Raw base58
recipient → rejected even when valid (models mistype; injected text smuggles
addresses). Over-cap amount → rejected with the cap in the message. Unknown
program in an inspected transaction → flagged, never guessed. Tests assert
policy rejections make **zero RPC calls**, and the happy path re-parses the
produced transaction and verifies Squads instruction bytes, PDAs, and the
embedded amount. Anchor discriminators are re-derived from a real sha256 of
the Squads v4 instruction names, and a fixture harness is ready to pin the
encoders byte-for-byte against a proposal from the official Squads app (the
captured fixture lands before submission). A host-smoke harness loads the built
components through the real ZeroClaw wasmtime adapter — metadata probe,
flat-config injection, policy refusals, and offline decode all pass on the
production code path.

## Engineering depth

`quorum-squads-core`: a from-scratch, wasm32-wasip2-friendly Solana substrate
(no solana-sdk, no async runtime): base58, sha256 PDAs with a real ed25519
off-curve check, compact-u16 legacy message compilation, the Squads v4
SmallVec wire format, Anchor discriminators, borsh account decoders, SPL +
Token-2022 builders, exact integer amount parsing, and token-capped receipt
shaping. Ground truth was vendored from the Squads v4 program source, and
test vectors were generated by an independent pure-Python implementation, so
the Rust is verified against something it did not produce. Published on
crates.io as shared infrastructure for any future Solana plugin.

## Judging criteria mapping

- **Security/custody**: spend authority is structurally absent, not capped.
- **Real-world usefulness**: a treasury desk any DAO/team can run today —
  propose from Telegram, approve in Squads mobile.
- **Ecosystem contribution**: three registry plugins + a published crate +
  an upstream-shaped PR with rollback path.
- **Roadmap**: Squads on-chain SpendingLimits — the only T2 design we know
  of where the cap survives total model compromise.
